# 家庭点餐微信小程序：开发实施文档

本文档用于指导后续实际开发。面向 Codex 和开发执行者，目标是把产品需求拆成可以落地的微信小程序项目结构、页面、数据库、云函数和阶段任务。

## 1. 开发目标

第一阶段先完成一个可在微信开发者工具中运行的本地演示版。

技术路线：

- 微信原生小程序
- 微信云开发
- 微信原生组件
- 云数据库
- 云存储
- 云函数

第一阶段不做：

- 正式上线审核
- 支付
- 配送
- 自建服务器
- 域名备案
- 重库存
- 批量导入菜谱

## 2. 项目配置

开发前需要用户提供：

```text
小程序 AppID：
云开发环境 ID：
小程序名称：
第一个家庭名称：
主理人昵称：
测试成员昵称：
```

敏感信息处理规则：

- 不需要 AppSecret。
- 不把账号密码、支付密钥、私钥写入项目。
- 云开发环境 ID 可以写入小程序配置。

建议项目结构：

```text
  pages/
  components/

cloudfunctions/
  login/
  createFamily/
  joinFamily/

  submitCart/

  updatePurchaseItem/
  addReview/
project.config.json
```

## 3. 用户角色与权限

系统角色只有两种：

### 主理人

主理人拥有全部权限：

- 创建家庭
- 邀请成员
- 管理菜品
- 上传菜品图片
- 编辑菜品食材配方
- 设置菜品状态
- 查看共享餐单
- 确认餐单
- 驳回菜品
- 修改建议采购量
- 划掉已有食材
- 生成采购单
- 查看统计

### 成员

成员权限：

- 通过邀请码加入家庭
- 浏览上架菜品
- 点菜
- 使用随机、盲盒、转盘点餐
- 查看当前共享餐单
- 查看采购单
- 勾选采购项已买
- 饭后评分、点赞、留言

成员禁止：

- 修改菜品
- 查看食材配方
- 修改家庭设置
- 修改成员权限
- 确认餐单
- 生成采购单

权限判断规则：

- 每次进入管理页面都校验当前用户是否为该家庭主理人。
- 成员端不展示主理人入口。
- 云函数也要做权限校验，不能只靠前端隐藏按钮。

## 4. 页面规划

### 公共页面

#### 登录/启动页

用途：

- 获取微信用户 openid。
- 判断用户是否已有家庭。
- 如果没有家庭，引导创建或加入。

主要状态：

- 未登录
- 已登录但未加入家庭
- 已加入家庭
- 多家庭切换

#### 家庭选择页

用途：

- 一个用户可能属于多个家庭。
- 主理人也可能管理多个家庭。

第一版可以做简单列表：

- 家庭名称
- 当前身份：主理人/成员
- 进入家庭

### 主理人页面

#### 主理人首页

入口模块：

- 今日共享餐单
- 菜品管理
- 采购单
- 历史记录
- 菜品统计
- 家庭成员

#### 创建家庭页

字段：

- 家庭名称

创建后：

- 当前用户成为该家庭唯一主理人。
- 生成邀请码。

#### 成员管理页

展示：

- 家庭邀请码
- 成员列表
- 成员昵称
- 加入时间

第一版不做复杂权限调整，因为一个家庭只有一个主理人。

#### 菜品管理页

展示：

- 菜品图片
- 菜名
- 分类
- 状态：上架、暂不可做、隐藏
- 总点餐次数

操作：

- 新增菜品
- 编辑菜品
- 切换状态

#### 新增/编辑菜品页

字段：

- 菜名
- 菜品图片
- 菜品分类
- 口味标签
- 状态
- 配方基准人数
- 食材配方

食材配方字段：

- 食材名称
- 数量
- 单位
- 备注
- 默认采购分类

注意：

- 食材配方只给主理人看。
- 成员点菜页不显示食材。

#### 餐单确认页

展示：

- 用餐日期
- 餐次
- 就餐人数
- 点菜成员
- 已点菜品
- 每道菜是谁点的

操作：

- 驳回菜品
- 填写驳回理由
- 查看系统建议食材量
- 修改采购量
- 划掉家里已有食材
- 修改采购分类
- 生成采购单

#### 采购单页

主理人和成员都可进入。

展示：

- 按采购分类分组
- 采购项名称
- 数量
- 单位
- 备注
- 是否已买
- 勾选人
- 勾选时间

操作：

- 勾选已买
- 取消勾选
- 手动添加额外物品

#### 菜品统计页

展示：

- 每道菜当前家庭点餐次数
- 主理人全部家庭合并点餐次数

第一版可先做简单列表，不做复杂图表。

### 成员页面

#### 加入家庭页

字段：

- 邀请码

提交后：

- 加入对应家庭。
- 身份为成员。

#### 成员首页

入口：

- 点菜
- 当前共享餐单
- 采购单
- 历史记录

#### 点菜页

布局：

- 左侧菜品分类
- 右侧菜品列表
- 底部购物车

菜品展示：

- 图片
- 菜名
- 口味标签
- 点餐次数

不展示：

- 食材配方
- 采购分类
- 成本信息

可点状态：

- 上架：可点
- 暂不可做：展示但不可点
- 隐藏：不展示

#### 购物车页

展示：

- 已选菜品
- 数量
- 删除
- 修改数量

提交时字段：

- 用餐日期
- 餐次
- 就餐人数

默认餐次规则：

- 05:00-10:00：今天早餐
- 10:00-12:00：今天午餐
- 12:00-15:00：今天下午茶
- 15:00-18:00：今天晚餐
- 18:00-05:00：次日早餐

提交后：

- 同一家庭、同一日期、同一餐次自动合并到同一共享餐单。

#### 共享餐单页

展示：

- 当前餐次
- 所有成员已经点的菜
- 每道菜是谁点的
- 当前状态

状态：

- 草稿中：成员可继续加菜
- 已生成采购单：锁定，不可改菜
- 已买齐
- 已完成

#### 趣味点餐页

第一版包含：

- 随机抽菜
- 盲盒点餐
- 转盘点餐

规则：

- 抽到菜以后，成员先确认。
- 确认后再进入购物车。
- 未确认不进入购物车。

#### 饭后评价页

字段：

- 评分
- 点赞
- 标签
- 留言

评价归属：

- 归属于对应家庭、日期、餐次的餐单。

## 5. 云数据库设计

以下为建议集合。

```js

  _id,

  nickname,

  createdAt,
  updatedAt

```

```js

  _id,
  name,
  ownerUserId,
  inviteCode,
  createdAt,
  updatedAt
}
```

说明：

- ownerUserId 是主理人用户 ID。
- 一个家庭只允许一个 ownerUserId。

```js
  _id,
  familyId,
  userId,
  role, // owner | member
  nickname,
  joinedAt
}
```

说明：

- 主理人在这里也有一条成员记录。
- 成员 role 为 member。

### chef_recipes 主理人个人菜谱表

```js
{
  _id,
  ownerUserId,
  name,
  imageFileId,
  category,
  tags,
  status,
  baseServings,
  ingredients,
  totalOrderCount,
  createdAt,
  updatedAt
}
```

ingredients 示例：

```js
[
  {
    name: "番茄",
    amount: 2,
    unit: "个",
    note: "选熟一点的",
    purchaseCategory: "蔬菜区"
  }
```

### family_dishes 家庭菜单表

```js
{
  _id,
  familyId,
  recipeId,

  name,

  status,
```

说明：

- family_dishes 是家庭内可点的菜单。

## 7. 关键业务算法

### 默认餐次推断

根据当前本地时间推断：

```text
05:00-10:00：今天早餐
10:00-12:00：今天午餐
12:00-15:00：今天下午茶
15:00-18:00：今天晚餐
18:00-05:00：次日早餐
```

如果当前时间为 18:00 到次日 05:00：

- mealDate 默认次日。
- mealType 默认 breakfast。

### 食材建议量计算

每道菜有 baseServings。

计算方式：

```text
建议数量 = 配方数量 × 本次就餐人数 ÷ 配方基准人数 × 菜品数量
```

示例：

- 番茄炒蛋配方适合 2 人。
- 配方需要番茄 2 个。
- 本次 4 人吃。
- 点 1 份番茄炒蛋。
- 建议番茄数量 = 2 × 4 ÷ 2 × 1 = 4 个。

注意：

- 单位不同不自动合并。
- 名称相同但单位不同，分开展示。
- 主理人可以手动修改。

### 采购项合并

可以自动合并的条件：

- 食材名称相同
- 单位相同
- 采购分类相同

不合并的情况：

- 名称相同但单位不同
- 名称相同但备注明显不同
- 主理人手动要求分开

### 点餐次数统计

统计时机：

- 主理人生成采购单后。

计数规则：

- 被确认的菜计数。
- 被驳回的菜不计数。
- 加入购物车但未提交不计数。
- 提交后但未确认不计数。

需要更新：

- family_dishes.familyOrderCount
- chef_recipes.totalOrderCount

## 8. 分阶段开发任务

### 阶段 0：项目初始化

任务：

- 创建微信原生小程序项目。
- 配置 AppID。
- 接入云开发环境 ID。
- 初始化 app.js、app.json、app.wxss。
- 建立页面目录。
- 建立云函数目录。
- 验证微信开发者工具可以运行。

验收：

- 小程序能在开发者工具启动。
- 控制台能成功初始化云开发。

### 阶段 1：登录、家庭、权限

任务：

- 实现 login 云函数。
- 实现用户表。
- 实现创建家庭。
- 实现邀请码加入家庭。
- 实现家庭选择。
- 实现主理人/成员权限判断。

验收：

- 主理人能创建家庭。
- 成员能通过邀请码加入。
- 成员看不到主理人管理入口。

### 阶段 2：菜品管理

任务：

- 主理人新增菜品。
- 上传菜品图片到云存储。
- 编辑菜品基础信息。
- 编辑食材配方。
- 切换菜品状态。
- 同步主理人菜谱到家庭菜单。

验收：

- 主理人能录入完整菜品。
- 成员点菜页看不到食材配方。
- 隐藏菜不展示，暂不可做菜不可点。

### 阶段 3：点餐、购物车、共享餐单

任务：

- 成员浏览菜单。
- 加入购物车。
- 默认推荐用餐日期和餐次。
- 提交购物车。
- 同一家庭、同一日期、同一餐次自动合并。
- 共享餐单展示所有成员点的菜和点菜人。

验收：

- 多个成员点同一餐会合并为一份共享餐单。
- 主理人生成采购单前，成员可以继续补菜。
- 生成采购单后，成员不能继续修改。

### 阶段 4：主理人确认与采购单

任务：

- 主理人查看共享餐单。
- 驳回菜品。
- 系统计算建议采购量。
- 主理人修改采购量。
- 主理人划掉已有食材。
- 生成采购单。
- 更新点餐次数。

验收：

- 建议采购量计算正确。
- 同名不同单位食材不强行合并。
- 采购单生成后餐单锁定。
- 被驳回菜不计数。

### 阶段 5：采购协同

任务：

- 成员查看采购单。
- 按分类展示采购项。
- 勾选已买。
- 记录勾选人和时间。
- 所有项目完成后更新采购单状态。

验收：

- 多个成员都能勾选。
- 勾选记录准确。
- 全部买齐后状态变为已买齐。

### 阶段 6：趣味互动与评价

任务：

- 随机抽菜。
- 盲盒点餐。
- 转盘点餐。
- 饭后评分。
- 点赞。
- 留言。

验收：

- 抽到菜后需要成员确认才加入购物车。
- 评价归档到对应餐单。

## 9. 页面文案与命名统一

统一使用：

- 主理人
- 成员
- 家庭
- 菜单
- 共享餐单
- 购物车
- 采购单
- 已买
- 已买齐
- 暂不可做

避免使用：

- 大厨
- 食客
- 售罄
- 订单支付
- 下单付款

因为本系统是家庭协作，不是商业交易。
